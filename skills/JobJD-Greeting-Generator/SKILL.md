# Job JD 打招呼话术生成

这个 skill 只做**一件事**：把岗位 JD 与候选人信息压缩成**一条最终可直接复制**的招聘 IM 话术。

操作说明、输入示例、场景差异详见 [README.md](README.md)。本文件只列 **agent 必须遵守的规则**。

## 什么时候触发

- 用户说“帮我根据 JD 写打招呼话术”
- 用户给出 `jd_url` / `jd_text`，想生成 BOSS直聘、智联招聘、前程无忧等平台的沟通开场白
- 用户说“已读未回，帮我跟进一句”或“投递后没反应，帮我催一下”

不该用本 skill 的场景：

- 代替用户发送消息
- 登录招聘平台、批量投递岗位
- 编造简历经历、项目成果或联系方式
- 生成完整简历、自我介绍长文或求职信

## 不可妥协的规则

1. **优先使用 `jd_url`。** 只要用户提供了 `jd_url`，必须先尝试从 URL 获取 JD 信息，而不是直接忽略链接。
2. **必须使用分阶段抓取。** 顺序固定为：直接页面提取 → readability/third-party 文本提取 → 基于回收正文做字段推断。
3. **只有在多路提取后仍不可靠时，才可以 ask for `jd_text`。** 不允许第一次抓取失败就直接让用户补文本。
4. **用户手填字段优先级最高。** `position_name`、`company_name` 以及其他显式补充信息，一旦用户提供，必须覆盖自动抽取结果。
5. **推断要保守。** 无法可靠判断 `position_name` / `company_name` 时，可以留空，不能硬猜。
6. **默认只输出最终话术。** 除非用户明确要求分析或调试说明，否则只返回 1 条最终可复制版本。
7. **默认只生成 1 个版本。** 不要在未被要求时给出 3 条、5 条候选话术。
8. **不得捏造候选人经历。** 只能基于用户提供的 `candidate_brief`、`candidate_highlights`、`candidate_status` 重组表达，不捏造项目、数据、学历或工作经历。
9. **不得施压或指责 HR。** 禁止生成“为什么不回复”“请尽快回复”“必须今天答复”之类带压迫感的话术。
10. **遵守平台约束。** 若平台不适合出现联系方式、夸张承诺或明显营销语，就不要生成这些内容。
11. **`first_contact` 的首句必须直入价值。** 默认骨架是：问好 + 年限/方向 + 最强优势，不要先铺垫岗位或公司信息。
12. **禁止默认使用岗位寒暄式开头。** 不要写“看到贵司正在招聘……”“关注到贵司这个岗位……”这类开头。
13. **禁止默认使用单独姓名自我介绍开头。** 不要先写“我是[姓名]”，除非用户明确要求保留这种风格。
14. **默认结尾必须短。** 优先使用“以下是我的简历，盼回复。”“简历附上，盼您回复。”这类 resume-forward close，不叠加“感谢查看”“祝工作顺利”等长尾。

## 输入规则

至少需要以下之一：

- `jd_url`
- `jd_text`

可选字段：

- `position_name`
- `company_name`
- `candidate_brief`
- `candidate_highlights`
- `candidate_status`
- `hr_role`
- `hr_gender`
- `tone_style`
- `scenario`
- `length_limit`
- `platform`

当 `jd_url` 与 `jd_text` 同时存在时：

1. 先从 `jd_url` 解析正文与页面显式字段
2. 再用 `jd_text` 补齐缺失信息或纠正明显偏差

## 抽取流程

### 第 1 步：直接页面提取

优先尝试读取原始页面，提取：

- 页面标题
- 主标题
- 公司信息区块
- JD 正文

### 第 2 步：readability / third-party 提取

如果直接页面结果为空、过短、只有壳内容、前端渲染不完整，或明显拿不到正文，则继续走 **readability / third-party** 文本提取。

这里的 third-party 仅用于**尽最大努力拿到可读正文**，不是比原页面更权威的数据源。

### 第 3 步：基于正文做保守推断

如果已经拿到可读 JD 文本，但结构化字段不完整，则可以基于正文推断：

- `position_name`
- `company_name`
- 职责关键词
- 技能关键词

推断不稳时宁缺毋滥。

### 第 4 步：向用户要 `jd_text`

只有 direct extraction 与 readability/third-party extraction 都失败，或回收内容仍明显不可靠时，才提示用户补充 `jd_text`。

## 字段优先级

固定优先级如下：

1. 用户手填字段
2. 原页面显式字段
3. readability/third-party 回收出的显式字段
4. 基于 JD 正文推断的字段

低可信字段不要硬塞进最终话术。

## 生成规则

内部统一按三段式组织：

1. **身份标签**：我是谁、做什么、当前状态如何
2. **岗位匹配点**：和 JD 最贴近的 2-3 个关键词，以及 1-2 条真实亮点
3. **礼貌收尾**：低压力、轻邀请式收尾

但 `first_contact` 的默认表面结构必须收紧成：

1. **问好**
2. **年限/方向 + 最强优势**
3. **极简动作型收尾**

可直接理解为：**问好 + 年限/方向 + 优势**，而不是“看到贵司招聘什么岗位 + 我是谁 + 我想沟通一下”。

### 场景规则

- `first_contact`：首次打招呼，重点是问好 + 年限/方向 + 最强匹配点 + 极简收尾
- `follow_up_read`：承接前一次联系，轻补充价值，不表达埋怨
- `follow_up_silent`：礼貌确认进度，不提“已读”，给对方留台阶

`first_contact` 额外强规则：

- 首句优先写能力和匹配度，不先解释“我为什么会来联系你”
- 若需要提岗位名，放后半句或省略，不抢开头
- 若 `candidate_brief` 信息不足，也优先保留方向和技能优势，不要拿空泛自我介绍凑字数

### 语气规则

- `正式`：更书面、更稳妥
- `自然专业`：默认，最通用
- `略活泼`：轻微口语化，但仍保持职业感，不使用表情

### 长度规则

默认遵守 `length_limit`，通常首次联系默认 100 字。

超过上限时按顺序压缩：

1. 删重复信息
2. 删弱修饰词
3. 缩短成果细节但保留关键词
4. 至少保留：身份标签 + 2 个核心匹配点关键词 + 礼貌收尾

## 输出规则

默认输出：

- 只输出 1 条最终话术

默认风格要求：

- 首句必须体现：**问好 + 年限/方向 + 最强优势**
- 不要默认出现“看到贵司正在招聘……”
- 不要默认出现单独的“我是[姓名]……”
- 结尾优先使用“以下是我的简历，盼回复。”或“简历附上，盼您回复。”这样的短尾
- 若已经用短尾表达动作，就不要再叠加“感谢查看”“祝工作顺利”“如方便的话”之类礼貌层

仅在用户显式要求解释时，才额外提供：

- 身份标签来源
- JD 匹配点
- 所用场景与语气

## 失败处理

- 若 `jd_url` 抓不到正文，不要立即停止；继续走 staged fallback
- 若多路提取后仍缺少可用 JD，则明确告诉用户当前缺口，并要求补 `jd_text`
- 若用户的候选人信息很少，就生成更短、更保守的话术，不得用想象补全
