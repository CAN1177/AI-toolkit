# 训练日志模板

> 本文件是**模板种子**。真实日志写在外部记录仓库的 `training-logs/<YYYY-MM-DD>-<problem-slug>.md` 中。

一题一文件。字段设计参考了 [My Google Recruitment Journey](http://blog.dominikrudnik.pl/my-google-recruitment-journey-part-1) 中关于"识别模式、压缩成触发信号、用自己的风格重写"的观察。

---

## 单题日志

```markdown
# <题目标题> · <YYYY-MM-DD>

## 元信息
- 难度：Easy / Medium / Hard
- 所属模式：<主模式>（可多选次模式）
- 来源：LeetCode #<编号> / 其他
- 语言：<go / py / ts / ...>

## Timebox
- 分配时长：<N> 分钟
- 实际用时：<N> 分钟
- 卡点出现在第几分钟：<N>
- 是否需要 LLM 提示：否 / 概念提示 / 完整解法

## 核心不变量（这道题剥到最后在考什么常量关系？）
<用一句话写清这题背后那个不变量。例如：
 "窗口内字符计数的总和恒等于窗口长度" →  滑动窗口;
 "前后缀乘积的交集 = 除自身以外的乘积" → 前缀积;
 "相同高度下，宽度越大水越多" → 对撞双指针>

## 攻击向量（看到这题时你第一个尝试的切入方式）
<例如：先暴力 O(n²) 验证思路，再找重复计算点 → 哈希优化>

## 模式 + 为什么这个模式适用
<写出最终选定的模式，并用一句话解释它为什么契合上面的"核心不变量"。>

## 最小骨架（5-10 行伪代码，能背下来的那种）
```
<写一份脱离本题的、可复用的代码骨架。不是完整解答，是"下次遇到同类问题我直接往里填"的那版>
```

## 我的写法 vs 最优解（差异对照）
| 维度 | 我的写法 | 最优解 | 差异原因 |
| :--- | :--- | :--- | :--- |
| 时间复杂度 | | | |
| 空间复杂度 | | | |
| 控制流 | | | |
| 命名 / 变量组织 | | | |

> 不盲抄最优解，但要清晰看到差距。下一次带着这个差距再练一道同族题。

## 触发信号（下次在什么"气味"下应该想到这个模式？）
<写 2-3 条具体到能让未来的你一看就激活的 trigger。例如：
 - "题目要求 O(1) 查询某元素是否出现过" → 哈希表
 - "数组有序 + 要找某种极值" → 二分或双指针
 - "要在无向图里判断能否走到 B" → BFS/DFS>

## 关键失误
<今天最大的 bug / 弯路 / 边界漏掉。越具体越好。>

## 口头讲解版本（不含代码，只用生活化语言 2-3 分钟讲清这题）
<想象你在对不懂算法的朋友解释。如果能讲清，说明你真的懂了。
 压力场景下，这段文字比代码更容易在脑子里重建。>

## 下一题建议
<一道邻近题或邻近模式，由教练给出或自己挑。只写一道，不贪多。>
```
