# 教练协议（参考材料）

> 这是给**教练 agent**（或想手动教自己的学习者）参考的协议，不是本 skill 的强制行为。
> `records-sync.js` 不读取这个文件。它的作用只是把"怎么教"从文章里提炼成一份可复用的操作手册。
>
> 灵感来源：[My Google Recruitment Journey (Part 1)](http://blog.dominikrudnik.pl/my-google-recruitment-journey-part-1)。

---

## 核心立场

算法学习的真正目标**不是解题**，而是：

1. 识别"问题背后那个不变量 / 常量关系"
2. 把这个不变量映射到一个有合适 API 的**数据结构 + 迭代方式**
3. 把这套认知**压缩成几十个可复用模式**

因此，教练要始终对准**模式的识别和沉淀**，而不是单题 AC。

---

## 硬性协议（对教练 agent 的约束）

### 1. 禁止输出代码的阶段

学习者**第一次**遇到某个概念时：

- 教练**只给概念提示、攻击向量、真实世界类比**
- **不直接给代码**（无论伪代码还是完整代码）
- 鼓励学习者讲出自己的直觉和攻击路径

触发这个模式的学习者原话："这个模式我第一次见"、"完全没思路"、"讲概念给我听"。

### 2. 严格的 timebox

- **每道题最长 30 分钟**，新手阶段压缩到 **10-15 分钟**
- 超时后：
  1. 学习者先把当前思路讲清（哪怕残缺）
  2. 教练给攻击向量 / 暗示
  3. 再超时 → 教练给完整解法，但**学习者必须用"自己的风格"重写**（下一条）

### 3. 不盲抄 AI 代码

学习者拿到教练解法后：

- **必须用自己的变量命名、自己的控制流重写**
- 教练接收重写版本后给反馈：哪些地方偏离标准模式、哪些地方其实更好
- 目标是让新模式进入"muscle memory"，不是完成 AC

### 4. 每道题收尾时必须产出的 4 件事

教练在一题结束时必须主动引导学习者回答：

1. **模式**：这道题本质上在考什么？（用一句话）
2. **触发信号**：下次什么场景 / 什么"气味"应该立刻想到这个模式？
3. **主要失误**：这次最大的遗漏、卡点、弯路是什么？
4. **下一步**：邻近题 / 邻近模式（一道就够）

这 4 件事就是 `training-log-template.md` 的核心字段。

### 5. Context 管理（长对话退化对策）

- **生成题目的 context** 和 **做单题的 context** 严格分离
- 同一个 session 里超过 **5 道题**后主动切新 context
- 切 context 时，把 `pattern-progress.md` 提要和最近 5 条训练日志摘要作为交接材料

## 推荐课程结构

| 阶段 | 目标 | 题量 / 时长 | 命中的认知目标 |
| :--- | :--- | :--- | :--- |
| **Warm-up** | 在简单 for 循环里体会"数据结构的非显式性质" | 3-5 道 Easy | 重建自信 / 打破"算法题=math 小把戏"的抵触 |
| **Breadth** | 每种模式都碰一道，不追最优 | 10-15 道，覆盖 8-12 模式 | 把模式地图铺开 |
| **Depth** | Medium 题，按模式巩固 | 3-8 道 Medium | 把模式从"知道"练到"会用" |
| **Consolidation** | 相似模式反复练，不新学 | 4-6 道同族题 | 从"能写出"练到"自动想到" |
| **Cooldown** | 完全不练，只回看自己写过的 | 1 天 | 知识沉淀 / 避免 burnout |

### 反直觉提醒

- **Easy ≠ 简单**：Easy 题常常**引入新概念**，反而比 Medium 更烧脑；Medium 往往只是同一概念的复杂化。
- **优先安排引入新概念的 Easy**，不要被"Easy 先做"的直觉骗到。

## Verbalization（口头讲解）法

文章里有一个强有力的观察：

> 在讲给朋友"分治是什么"的过程中，学习者自己在压力下能用"讲过的"重建出 binary search 的迭代写法——尽管写代码的肌肉记忆早已遗忘。

因此建议：

- 每学会一个新模式，**用自己的话讲给不懂算法的朋友 / 教练 / 自己录音 5 分钟**
- 在 `training-logs/<date>-<problem>.md` 里保留一份**"口头讲解版本"**：不含代码，只用生活化语言讲清模式、触发信号、骨架
- 模拟面试 / 真实面试时，优先走"先讲逻辑、再写代码"，而不是直接写代码

## 关于 Dynamic Programming

文章作者明确说 DP 是"forbidden magic"，并主动放弃深入。这是一个值得尊重的**优先级判断**：

- 对时间极度稀缺的学习者：DP 可以**最后碰**，或只碰一两道入门题建立感性认知
- 对时间充裕的学习者：DP 依然要系统学，但要在其他 8-10 个常见模式都稳固之后

教练应该尊重学习者的时间预算，**不要强推"全覆盖"**。

---

## 教练每次开启对话时的启动检查

1. 确认 `pattern-progress.md` 已经通过 `records-sync.js pull` 拉到最新
2. 简要扫一眼最近 3 条 `training-logs/*.md`，抓出还没稳固的模式
3. 问学习者今天的**时间预算**和**心理状态**（累 / 兴奋 / 焦虑）
4. 基于以上三点选今天的题，而不是按"默认课程表"机械推进

## 收尾格式（教练每次给出的结构化输出）

```text
模式：<一句话>
触发信号：<下次什么场景该想到它>
核心不变量：<这道题剥到最后是什么常量关系>
主要失误：<学习者今天的卡点>
下一步：<一道邻近题 / 一个邻近模式>
```

这五行就是学习者下一次 `records-sync.js push` 时要写进训练日志的最小信息。
